# Results — Draft

## Overview

Initial evaluation using the synthetic dataset generated by `traffic_gen`. The dataset contains
100 packets: 40 legit and 60 malicious, interleaved in a 4:6 ratio per 10-packet block.

---

## Dataset

| Category | Count | Description |
|---|---|---|
| Legit HTTP (port 80) | 16 | GET, POST, PUT, HEAD requests with valid Host headers |
| Legit HTTP (port 443) | 8 | GET and POST requests on HTTPS port |
| Legit DNS (UDP port 53) | 16 | Valid DNS A-record queries with correct header structure |
| Malicious — HTTP on DNS port | 12 | HTTP GET/POST payloads sent to UDP port 53 |
| Malicious — DNS on HTTP port | 12 | Binary DNS headers sent to TCP port 80 |
| Malicious — bad HTTP method | 6 | Unrecognized method verb (e.g., `BADVERB`) on port 80 |
| Malicious — missing Host header | 6 | HTTP request missing required Host: field |
| Malicious — empty payload | 6 | Zero-byte payload on port 80 |
| Malicious — missing HTTP version | 6 | Request line with no HTTP/1.x version token |
| Malicious — truncated DNS | 12 | 4-byte DNS header (below 12-byte minimum) on UDP port 53 |

## Initial Results (Synthetic Dataset)

| Metric | Value | Target |
|---|---|---|
| Detection rate (TP / (TP + FN)) | 100% | ≥ 85% |
| False positive rate (FP / (FP + TN)) | 0% | ≤ 15% |

All 60 malicious packets were correctly flagged. All 40 legit packets were correctly cleared.

---

## Observations

**Protocol impersonation is reliably detected.** HTTP payloads on UDP port 53 fail immediately
at the DNS validator's label-parsing step — the HTTP text does not form a valid null-terminated
DNS name. DNS payloads on TCP port 80 fail the HTTP method check since the binary DNS header
does not start with a valid ASCII method string.

**Malformed HTTP is reliably detected.** The HTTP validator's ordered rule chain catches each
attack type at a specific rule:
- Invalid method → caught at Rule 4 (method check)
- Missing Host → caught at Rule 6 (Host header check)
- Empty payload → caught at Rule 3 (payload length check)
- Missing HTTP version → caught at Rule 5 (HTTP version string check)
- Truncated DNS → caught at Rule 3 (DNS header minimum length)

**No false positives on the synthetic legit set.** Valid HTTP GET/POST/PUT/HEAD with Host
headers pass all 7 rules. Valid DNS queries with correct header structure pass all 5 rules.

## Limitations & Next Steps

The current dataset is entirely synthetic and designed to cover the specific attack variants
implemented in `traffic_gen.c`. The 100% detection rate reflects deterministic rule coverage
against known attack patterns — it does not yet reflect performance against real-world or novel
traffic.

Week 16 evaluation will:
1. Add captures from `curl` and `dig` to validate against real protocol behaviour
2. Expand the malicious set with additional variants (e.g., partial HTTP headers, DNS with
   compression pointers, HTTP/2 framing)
3. Generate detection rate / FP rate charts from `metrics.csv`
4. Report final results comparing observed rates against the ≥ 85% / ≤ 15% targets
